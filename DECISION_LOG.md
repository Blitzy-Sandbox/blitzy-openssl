# Decision Log — OpenSSL C → Rust Migration

> **Document purpose:** This is the **single source of truth** for every non-trivial
> architectural and implementation decision made during the OpenSSL C-99 → Rust
> (edition 2021) rewrite. Rationale lives here, not in code comments. Any deviation
> from the literal interpretation of requirements has an explicit entry.
>
> **AAP Reference:** §0.8.5 (Explainability Rule)
>
> **Format:** Markdown table — columns: #, Decision, Alternatives Considered,
> Rationale, Risks, Date.
>
> **Scope:** All non-trivial decisions across the entire workspace — architecture,
> dependencies, tooling, conventions, deviations, and trade-offs.

## Architecture Decisions

| # | Decision | Alternatives Considered | Rationale | Risks | Date |
|---|----------|------------------------|-----------|-------|------|
| A-1 | **Workspace architecture: 7-crate split** (`openssl-common`, `openssl-crypto`, `openssl-ssl`, `openssl-provider`, `openssl-fips`, `openssl-cli`, `openssl-ffi`) | (a) Monolithic single crate; (b) 3-crate split (lib/cli/ffi); (c) Per-algorithm micro-crates (~50 crates); (d) 4-crate split (crypto/ssl/cli/ffi) without separate common/provider/fips | The 7-crate layout directly mirrors the C build artifact boundaries (`libcrypto.so`, `libssl.so`, FIPS module, CLI binary) while adding three Rust-specific separations: `openssl-common` breaks circular dependencies between crypto/ssl/provider, `openssl-fips` enforces FIPS certification isolation via crate visibility, and `openssl-ffi` confines all `unsafe` to a single crate per Rule R8. The monolithic approach was rejected because it prevents independent compilation of the FIPS module. Per-algorithm micro-crates were rejected due to excessive inter-crate boilerplate for ~50+ algorithm directories. See `ARCHITECTURE.md` §1 for full boundary justifications. (AAP §0.4.1, §0.7.3) | Cross-crate dependency management complexity; compile time increases with crate count; potential for type duplication across boundaries; version coordination overhead during evolution | 2026-04-09 |
| A-2 | **Provider dispatch: Rust trait objects (`Box<dyn Provider>`)** replacing C `OSSL_DISPATCH` function pointer tables | (a) Preserve C-style raw function pointer tables with `unsafe`; (b) Enum dispatch (static, exhaustive match); (c) Generics with monomorphization (static dispatch only); (d) `enum_dispatch` crate for hybrid approach | Trait objects preserve the runtime provider loading semantics that are fundamental to OpenSSL's architecture — providers are selected at runtime via property queries and can be loaded from shared libraries. Raw function pointer tables (alternative a) require `unsafe` at every call site, violating Rule R8. Enum dispatch (alternative b) requires recompilation to add providers, breaking the extensibility model. Static generics (alternative c) prevent runtime selection. The C `OSSL_DISPATCH` pattern (function ID + function pointer pairs, iterated at load time) maps naturally to Rust trait objects with dynamic dispatch. Source evidence: `crypto/provider_core.c` line 198 (`const OSSL_DISPATCH *dispatch`) stores dispatch tables per provider; the Rust equivalent is `Arc<dyn Provider>`. (AAP §0.7.1) | Dynamic dispatch overhead (~1-3ns per virtual call); trait object limitations (no associated types without `dyn`-compatible workarounds); object safety constraints require careful trait design; loss of monomorphization optimization on hot paths | 2026-04-09 |
| A-3 | **`OSSL_PARAM` system → strongly-typed Rust config structs** with `ParamSet` bridge for dynamic dispatch | (a) `HashMap<String, Box<dyn Any>>` (fully dynamic); (b) Preserve raw parameter bags with string keys; (c) `serde_value::Value` enum; (d) Typed structs only (no dynamic bridge) | The C `OSSL_PARAM` system (`crypto/params.c`) is a dynamically-typed parameter passing mechanism with ~400 unique parameter names, identified by string keys and type-cast at runtime. Strongly-typed Rust structs (e.g., `AesCipherParams { key: Vec<u8>, iv: Option<Vec<u8>>, padding: PaddingMode }`) move validation from runtime string matching to compile-time type checking, eliminating an entire class of type confusion bugs. The `ParamSet` bridge (`HashMap<&'static str, ParamValue>` where `ParamValue` is a typed enum) preserves dynamic dispatch for providers that need runtime parameter negotiation. Alternative (a) loses all type safety. Alternative (d) prevents provider extensibility. (AAP §0.7.2) | Less flexibility for third-party provider parameters that define custom keys at runtime; migration effort to define typed structs for all ~400 parameter names; potential for struct proliferation; bridge layer adds indirection | 2026-04-09 |
| A-4 | **Async runtime: tokio for QUIC stack only**, sync everywhere else. Single `Runtime` instance per Rule R1 | (a) `async-std` instead of tokio; (b) `smol` lightweight runtime; (c) Fully synchronous (poll-based, matching C implementation); (d) Async everywhere (all crates use tokio); (e) No async — use `mio` directly for QUIC | The QUIC v1 reactor pattern (`ssl/quic/quic_engine.c`) is inherently event-driven with concurrent stream processing, timer management, and I/O multiplexing — a natural fit for async/await. Tokio is selected over alternatives for its LTS commitment (tokio 1.x LTS until March 2027), extensive ecosystem (`tokio-util` for codec/framing), and proven production track record. Fully sync (alternative c) would require reimplementing a custom event loop. Async everywhere (alternative d) would force `tokio` as a dependency on `openssl-crypto`, unnecessarily burdening consumers who only need cryptographic primitives. The `openssl-ssl` crate is the sole owner of async code; `openssl-crypto` remains fully synchronous with zero tokio dependency. Exactly one `Runtime::new()` call exists (in `openssl-cli::main()`); all other components receive a `Handle`. (AAP §0.7.4, Rule R1) | Sync/async boundary complexity at the `openssl-ssl` ↔ `openssl-crypto` interface; non-deterministic task scheduling in QUIC (vs deterministic C poll loop); risk of accidental `.await` across sync mutex holds (mitigated by `clippy::await_holding_lock = "deny"` per Rule R2); blocking calls in async context require `spawn_blocking` bridge | 2026-04-09 |
| A-5 | **Error handling: `thiserror` derive with per-crate error enums** (`CryptoError`, `SslError`, `ProviderError`, `FipsError`) | (a) `anyhow` for all errors; (b) `eyre` with color-eyre; (c) Raw `Box<dyn std::error::Error>`; (d) `error-chain` (deprecated); (e) Single unified error enum for entire workspace | Per-crate error enums with `thiserror` derive provide structured, type-safe error handling that replaces the C `ERR_*` thread-local error stack. Each error variant carries structured data (algorithm name, key size, reason code) rather than opaque integer codes. Error chaining via `#[from]` and `std::error::Error::source()` replaces the C error queue's linked list. `anyhow` (alternative a) was rejected for library crates because it erases error types, preventing callers from matching on specific error conditions — this is critical for FIPS indicator checks and protocol state machine error recovery. `eyre` (alternative b) was rejected for the same reason. A single unified enum (alternative e) was rejected because it would create a massive enum with hundreds of variants spanning unrelated subsystems. (AAP §0.7.7) | Error enum proliferation as subsystems grow; cross-crate error conversion boilerplate via `From` impls; potential for enum variant explosion in `CryptoError` given 50+ algorithm families; loss of the C error stack's "append additional context" pattern (partially mitigated by `tracing::Span` context) | 2026-04-09 |
| A-6 | **FIPS module as independent crate** (`openssl-fips`) with crate-visibility enforcement | (a) Feature flag on `openssl-crypto` (`--features fips`); (b) Separate Cargo workspace entirely; (c) Conditional compilation within `openssl-provider` (`#[cfg(fips)]`) | The FIPS provider must be independently certifiable under NIST CMVP, requiring strict isolation from non-approved algorithms. A dedicated crate enforces this isolation through Rust's crate visibility system — private items cannot leak across crate boundaries, providing a compiler-enforced boundary that is stronger than C's `#ifdef FIPS_MODULE` conditional compilation. Source evidence: `providers/fips/fipsprov.c` line 38 defines `FIPS_DEFAULT_PROPERTIES[] = "provider=fips,fips=yes"`, showing the C isolation model. The crate approach enables independent `cargo test -p openssl-fips` without pulling in non-FIPS code. A feature flag (alternative a) was rejected because feature flags are additive in Rust — a dependency could accidentally enable the FIPS feature, pulling in non-approved code paths. A separate workspace (alternative b) was rejected because it prevents workspace-level lint inheritance and complicates CI. (AAP §0.7.3) | Code duplication between FIPS-approved and default provider implementations for shared algorithms (AES, SHA, RSA); FIPS crate must carefully re-export only approved algorithms from `openssl-crypto`; testing matrix doubles (FIPS + non-FIPS); independent certification timeline may lag behind main crate updates | 2026-04-09 |
| A-7 | **BIO I/O abstraction → Rust `Read`/`Write`/`AsyncRead`/`AsyncWrite` traits** with `Bio` trait for filter chaining | (a) Direct port of BIO struct with function pointers; (b) Use only `std::io::Read`/`Write` without custom trait; (c) Create `Bio` wrapper around `tokio::io` only | The C BIO system (`crypto/bio/*.c`, 28 files) is an I/O abstraction with filter chaining (e.g., base64 BIO → buffer BIO → socket BIO). Rust's standard `Read`/`Write` traits provide the synchronous I/O foundation, while `tokio::io::AsyncRead`/`AsyncWrite` cover the QUIC async path. A custom `Bio` trait extends these with filter chain composition (analogous to `BIO_push`/`BIO_pop`) and type-safe method dispatch (replacing `BIO_ctrl` integer commands). Alternative (a) was rejected because it replicates C's function pointer unsafety. Alternative (b) was rejected because it lacks filter chaining. (AAP §0.4.1) | Custom `Bio` trait adds learning curve; impedance mismatch between sync `Read`/`Write` (used in TLS record layer) and async `AsyncRead`/`AsyncWrite` (used in QUIC); filter chain composition in async context requires careful lifetime management | 2026-04-09 |
| A-8 | **Handshake state machine: type-state pattern** encoding valid transitions at compile time | (a) Runtime enum with match-based transitions (direct C port); (b) Petri-net style state machine library; (c) `state_machine_future` crate | The C handshake state machine (`ssl/statem/statem.c`) uses a dual-layer design: MSG_FLOW (outer read/write loop) + HANDSHAKE (inner message processing). The type-state pattern encodes valid state transitions in Rust's type system, making invalid transitions a compile error rather than a runtime assertion failure. For example, `Handshake<ClientHello>` can only transition to `Handshake<ServerHello>` — calling a `ServerFinished` method on a `ClientHello` state is a type error. Alternative (a) was rejected because it defers transition validation to runtime, replicating the same assertion-failure failure mode as C. (AAP §0.4.3) | Type-state pattern increases generic parameter complexity; each state transition creates a new type, potentially impacting compile times; some transitions are inherently dynamic (e.g., HelloRetryRequest loops) and require runtime fallback; debugging type errors in deeply nested generics is challenging | 2026-04-09 |

## Dependency and Tooling Decisions

| # | Decision | Alternatives Considered | Rationale | Risks | Date |
|---|----------|------------------------|-----------|-------|------|
| D-1 | **BigNum: `num-bigint` crate** for arbitrary-precision integer arithmetic (replacing `crypto/bn/*.c`, 39 files) | (a) Custom bignum implementation ported from C; (b) `ibig` crate; (c) `rug` crate (GMP wrapper via `-sys`); (d) `malachite` crate; (e) `crypto-bigint` from RustCrypto | `num-bigint` is pure Rust (no C dependencies), well-tested, and widely used. `rug` (alternative c) was rejected because it wraps GMP via `gmp-mpfr-sys`, introducing a C dependency that contradicts the pure-Rust goal and would be flagged by `deny.toml`. `crypto-bigint` (alternative e) provides constant-time operations but requires fixed-size integers, which is incompatible with OpenSSL's variable-size bignum API. `ibig` (alternative b) is less mature. Custom implementation (alternative a) was rejected due to the high risk of subtle bugs in modular arithmetic — the C `bn/` subdirectory represents 39 files of battle-tested code. (AAP §0.6.1) | Performance gap for modular exponentiation and Montgomery multiplication — `num-bigint` lacks optimized assembly backends that the C BN code uses; potential bottleneck for RSA key generation and signing; may need to revisit with `crypto-bigint` for constant-time RSA if timing side-channel concerns arise | 2026-04-09 |
| D-2 | **ASN.1/X.509: RustCrypto ecosystem** (`der`, `x509-cert`, `pem-rfc7468`, `pkcs8`, `spki`) | (a) `yasna` crate; (b) `asn1-rs` crate; (c) Hand-rolled ASN.1 parser; (d) `rasn` crate | The RustCrypto `der` crate provides well-maintained ASN.1 DER encoding/decoding with seamless integration into the `x509-cert` crate for certificate parsing, `pkcs8` for private key encoding, and `spki` for public key info. This ecosystem provides interoperable types that share underlying DER primitives, reducing conversion boilerplate. `yasna` (alternative a) is less actively maintained. `asn1-rs` (alternative b) has a different API style that doesn't integrate with the RustCrypto key type ecosystem. Hand-rolling (alternative c) was rejected due to the complexity of the OpenSSL ASN.1 template system (`crypto/asn1/`, 65 files). (AAP §0.6.1) | Feature coverage gaps for OpenSSL's complex ASN.1 templates (e.g., `X509_ATTRIBUTE`, CMP structures); RustCrypto `der` crate may not support all implicit/explicit tagging patterns used in OpenSSL; potential need for custom extensions to handle legacy formats | 2026-04-09 |
| D-3 | **Secure memory erasure: `zeroize` crate** with derive macro for all key material types | (a) `secrecy` crate with `SecretString`/`SecretVec`; (b) Custom `Drop` implementations with `std::ptr::write_volatile`; (c) `memsec` crate; (d) Inline assembly volatile stores | `zeroize` is the Rust ecosystem standard for secure memory zeroing, providing a `Zeroize` derive macro that integrates cleanly with struct definitions — `#[derive(Zeroize, ZeroizeOnDrop)]` on key material types ensures automatic cleanup. It uses compiler-fence and volatile writes to prevent dead-store elimination. `secrecy` (alternative a) wraps values in `Secret<T>` which prevents accidental logging but adds wrapper overhead for every access. Custom `Drop` (alternative b) is error-prone and not guaranteed to survive compiler optimization. The `zeroize` crate directly replaces `OPENSSL_cleanse()` and `OPENSSL_secure_malloc()` patterns. (AAP §0.7.6) | Compiler optimizations may still theoretically elide zeroing in edge cases (mitigated by `zeroize`'s use of `core::sync::atomic::compiler_fence`); derive macro overhead at compile time; `Zeroize` trait adds a bound constraint to generic key material containers | 2026-04-09 |
| D-4 | **Constant-time operations: `subtle` crate** for comparison and conditional selection | (a) Hand-rolled constant-time comparisons (port of `include/internal/constant_time.h`); (b) `ct-codecs` crate; (c) Inline assembly for constant-time branches | The `subtle` crate provides `ConstantTimeEq`, `ConditionallySelectable`, and `Choice` types that enforce constant-time behavior at the type level. Using a well-audited crate eliminates the risk of accidentally introducing timing side channels in hand-rolled comparisons. The C codebase uses custom macros in `include/internal/constant_time.h` for this purpose. (AAP §0.7.6) | Dependency on external crate for security-critical primitive; `subtle` cannot prevent all timing leaks (e.g., cache-timing attacks from data-dependent memory access); must verify that compiler does not optimize away constant-time properties at higher optimization levels | 2026-04-09 |
| D-5 | **CLI framework: `clap` with derive macros** for 56+ subcommands | (a) `structopt` (deprecated, merged into clap); (b) `argh` by Google; (c) Hand-rolled argument parser; (d) `bpaf` crate | `clap` with derive macros provides declarative subcommand definition, automatic help generation, shell completions, and argument validation — all critical for replicating OpenSSL's complex CLI with 56+ subcommands. The derive API reduces boilerplate compared to the builder API. `argh` (alternative b) is minimal and lacks features needed for OpenSSL's complex option parsing (e.g., mutually exclusive groups, value parsing). Pinned to `=4.5.23` to avoid edition 2024 transitive dependency (see D-8). (AAP §0.6.1) | Pin to `=4.5.23` prevents receiving clap bugfixes and new features until MSRV is raised; clap's derive macro increases compile times; complex subcommand hierarchy may require custom value parsers for OpenSSL-specific option formats (e.g., cipher suite strings, key format specifiers) | 2026-04-09 |
| D-6 | **Observability stack: `tracing` + `metrics` + `metrics-exporter-prometheus`** shipped with initial implementation | (a) `log` crate with `env_logger`; (b) `slog` structured logging; (c) Defer observability to post-migration; (d) OpenTelemetry SDK directly | Observability is shipped with the initial implementation per the AAP Observability Rule — not as a follow-up. `tracing` provides structured logging with span-based correlation IDs and integrates with OpenTelemetry via `tracing-opentelemetry`. `metrics` provides a lightweight facade for counters, gauges, and histograms. `log` (alternative a) lacks structured spans and correlation ID propagation. Deferring (alternative c) violates the "ship observability with initial implementation" mandate. (AAP §0.8.5 Observability Rule) | Additional dependency weight for consumers who don't need observability; `tracing` subscriber initialization must happen early in program startup; potential performance impact from tracing instrumentation in hot crypto paths (mitigated by compile-time level filtering) | 2026-04-09 |
| D-7 | **Benchmarking: `criterion` crate** for Gate 3 performance baseline comparisons | (a) `std::hint::black_box` with manual timing; (b) `divan` crate; (c) `iai` (instruction-count based); (d) Custom benchmark harness | `criterion` provides statistical analysis (confidence intervals, regression detection), HTML report generation, and comparison across runs — all needed for the ±20% parity target in Gate 3. It is the de facto standard for Rust benchmarking. `iai` (alternative c) measures instruction counts rather than wall-clock time, which doesn't align with the "wall-clock + peak memory" requirement. (AAP §0.8.2, Gate 3) | Criterion benchmark results vary with system load; HTML report generation adds build complexity; `criterion` requires `cargo bench` (nightly-free since 0.5) but may have different results than production workloads | 2026-04-09 |

## Language Translation Decisions

| # | Decision | Alternatives Considered | Rationale | Risks | Date |
|---|----------|------------------------|-----------|-------|------|
| L-1 | **Assembly strategy: pure Rust initial implementation**, with `core::arch` intrinsics as fallback if ±20% parity target is not met | (a) Link existing perlasm-generated assembly via FFI; (b) Rust inline assembly (`core::arch::asm!`) for all hot paths; (c) Auto-vectorization only, no intrinsics; (d) Wrap existing `.S` files as a static library | The safety-first approach starts with pure Rust implementations that rely on LLVM auto-vectorization and `#[cfg(target_feature)]` guards. The 239 perlasm files (`crypto/**/asm/*.pl`) generate platform-specific assembly for AES-NI, SHA extensions, ARM NEON, AVX-512 — these represent years of hand-optimized code that cannot be trivially replicated. Pure Rust defers the complexity of inline assembly while establishing correctness. If benchmarks (Gate 3) reveal >20% regression on hot paths (AES-GCM, SHA-256, ChaCha20-Poly1305, P-256), targeted `core::arch` intrinsics can be introduced in dedicated `accel` submodules with `// INTRINSIC:` justification. Linking existing assembly (alternative a) introduces `unsafe` FFI calls outside the FFI crate, violating Rule R8. (AAP §0.7.5) | Significant performance gap expected on AES-GCM and SHA-256 hot paths where hand-tuned assembly uses CPU-specific instructions (AES-NI, SHA-NI, VAES); pure Rust P-256 scalar multiplication may be 3-5x slower than optimized assembly; iterative optimization may be needed post-initial delivery | 2026-04-09 |
| L-2 | **`Option<T>` for all nullable/unset values**, eliminating sentinel values (`0`, `-1`, `""`) per Rule R5 | (a) Preserve C sentinel patterns with documented conventions; (b) Use `Option<T>` only for pointer-equivalent fields; (c) Custom `MaybeUnset<T>` wrapper type | Sentinel values are a pervasive source of bugs in C — e.g., checking `if (timeout == 0)` cannot distinguish "not set" from "zero timeout". Rust's `Option<T>` provides a type-safe encoding of presence/absence that the compiler enforces via exhaustive matching. Every C function that returns `0` for failure and a non-zero value for success becomes `Result<T, E>`. Every config field that uses `-1` for "unset" becomes `Option<T>`. Alternative (a) was rejected because it perpetuates the bug class being eliminated. (AAP §0.8.1, Rule R5) | Widespread API surface change from C conventions; `Option<T>` adds 1 byte (or more with alignment) to struct size per field; nested `Option<Option<T>>` may arise in edge cases; existing C callers via FFI still expect sentinel values (handled by conversion in `openssl-ffi`) | 2026-04-09 |
| L-3 | **`try_from` / saturating arithmetic for all numeric conversions**, no bare `as` casts per Rule R6 | (a) Allow bare `as` casts with code review; (b) Use `as` casts with `#[allow]` annotations; (c) Custom `SafeCast` trait; (d) `num::cast` functions | Bare `as` casts silently truncate — e.g., `let x: u8 = large_u32 as u8;` loses data without warning. Enforced via `#[deny(clippy::cast_possible_truncation, clippy::cast_sign_loss, clippy::cast_possible_wrap)]` in the workspace `Cargo.toml`. All narrowing conversions must use `T::try_from(value)?` (fallible) or `.saturating_cast()` / `.clamp()` (intentional clamping with `// TRUNCATION:` justification). Alternative (a) was rejected because code review is not a reliable enforcement mechanism for a 572,000-LoC migration. (AAP §0.8.1, Rule R6) | Verbosity increase — `u32::try_from(usize_val)?` is more code than `usize_val as u32`; potential for excessive error propagation in hot paths; some legitimate truncations (e.g., extracting a byte from a word) require `#[allow]` with justification | 2026-04-09 |
| L-4 | **RAII/`Drop` for all resource lifecycle management**, replacing manual `*_new()`/`*_free()` pairs | (a) Explicit `close()`/`free()` methods without `Drop`; (b) `Drop` only for system resources, manual free for crypto objects; (c) Reference counting via `Rc`/`Arc` for all objects | Every C allocation pattern (e.g., `SSL_CTX_new()`/`SSL_CTX_free()`, `EVP_MD_CTX_new()`/`EVP_MD_CTX_free()`) becomes a Rust struct with `Drop` implementation. Secure types additionally derive `Zeroize`/`ZeroizeOnDrop` to ensure key material is cleared before deallocation. Source evidence: `ssl/quic/quic_engine.c` lines 26-52 show `ossl_quic_engine_new()` (allocation + init) paired with `ossl_quic_engine_free()` (cleanup + dealloc) — the Rust equivalent is a `QuicEngine::new()` → automatic `Drop::drop()`. `Arc` is used only where shared ownership is genuinely required (e.g., `SSL_CTX` shared across connections). | `Drop` ordering in complex struct hierarchies may cause use-after-free if fields reference each other (mitigated by Rust's borrow checker); `Drop` panics during unwinding cause abort; some C objects have non-trivial cleanup ordering that must be carefully replicated | 2026-04-09 |
| L-5 | **Feature flags for `OPENSSL_NO_*` compile-time gating** (`#[cfg(feature = "ec")]`, etc.) | (a) Always compile all algorithms; (b) Runtime feature detection only; (c) Separate crates per algorithm family | C uses `#ifndef OPENSSL_NO_EC` preprocessor guards to conditionally exclude algorithm families. Rust's feature flags (`#[cfg(feature = "ec")]`) provide the equivalent mechanism with Cargo-level dependency resolution. Features are additive and documented in each crate's `Cargo.toml`. Alternative (a) was rejected because it increases binary size for consumers who don't need all algorithms. Alternative (c) was rejected per decision A-1 (per-algorithm micro-crates). | Feature flag combinatorial explosion — testing all feature combinations is impractical; some feature interactions may introduce subtle bugs; downstream consumers must correctly specify feature flags | 2026-04-09 |
| L-6 | **`Arc<RwLock<T>>` with fine-grained locking** for shared mutable state, per Rule R7 | (a) Single global coarse lock; (b) Lock-free data structures (`crossbeam`); (c) `RefCell` with runtime borrow checking; (d) Message-passing channels only | The C codebase uses 5+ independent locks in `provider_core.c` alone (flag_lock, activatecnt_lock, opbits_lock, default_path_lock, store lock) with documented ordering rules (lines 59-118). The Rust translation preserves this fine-grained approach using `Arc<parking_lot::RwLock<T>>` for non-async contexts and `Arc<tokio::sync::RwLock<T>>` for async contexts (QUIC only). Every shared lock carries a `// LOCK-SCOPE:` annotation explaining what data it protects and why a coarse lock is insufficient. Alternative (a) was rejected because the C code explicitly documents why fine-grained locking is necessary. Alternative (c) is single-threaded only. (AAP §0.8.1, Rule R7) | Increased deadlock risk with multiple fine-grained locks (mitigated by documented lock ordering, matching C's precedence rules); `parking_lot` vs `std::sync` choice adds a dependency; lock contention profiling needed to validate granularity decisions | 2026-04-09 |

## Safety and Compliance Decisions

| # | Decision | Alternatives Considered | Rationale | Risks | Date |
|---|----------|------------------------|-----------|-------|------|
| S-1 | **`unsafe` confined exclusively to `openssl-ffi` crate** per Rule R8. All other 6 crates are 100% safe Rust | (a) Allow `unsafe` in `openssl-crypto` for performance-critical paths; (b) Allow `unsafe` in any crate with `// SAFETY:` justification; (c) Zero `unsafe` anywhere including FFI (sacrificing C interop) | The `openssl-ffi` crate is the sole boundary between safe Rust and C ABI consumers. Confining `unsafe` to one crate concentrates audit effort, simplifies formal verification scope, and ensures that the core cryptographic logic benefits fully from Rust's safety guarantees. Workspace lint `unsafe_code = "deny"` enforces this — only `openssl-ffi` overrides with `#![allow(unsafe_code)]`. Expected ~200-400 `unsafe` blocks in `openssl-ffi`, proportional to the number of public FFI functions exported. Each block carries a `// SAFETY:` comment. Verification: `grep -rn "unsafe" crates/ --include="*.rs" | grep -v "openssl-ffi"` returns zero matches. (AAP §0.7.8, Rule R8) | Performance-critical crypto paths cannot use `unsafe` optimizations (e.g., unchecked indexing for proven-bounds array access); all platform-specific intrinsics must be safe wrappers; FFI crate maintenance burden is high with 200-400 unsafe sites | 2026-04-09 |
| S-2 | **FIPS self-test: Known Answer Tests (KATs) run at module load time** via `fips::self_test::run_kats()` | (a) Run KATs lazily on first algorithm use; (b) Run KATs in a separate process; (c) Skip KATs in debug builds | NIST CMVP requires Power-On Self-Test (POST) execution before any cryptographic operation. The C implementation (`providers/fips/self_test.c`) runs all KATs during `OSSL_provider_init()`. The Rust equivalent runs KATs during `FipsProvider::new()`, blocking until all tests pass. The FIPS state machine transitions: `PowerOn → SelfTesting → Operational | Error`. Lazy execution (alternative a) was rejected because it violates the CMVP requirement that POST completes before any service is available. Skipping in debug (alternative c) was rejected because it could mask real failures. (AAP §0.7.3) | Module load time increases by KAT execution duration (~10-50ms); any KAT failure permanently locks the module in `Error` state requiring process restart; KAT test vectors must be kept in sync with algorithm implementations | 2026-04-09 |
| S-3 | **Banned crate list in `deny.toml`**: `openssl-sys`, `openssl`, `native-tls` | (a) Allow `openssl-sys` in test dependencies only; (b) No ban list, rely on code review; (c) Ban all `-sys` crates globally | Banning `openssl-sys`, `openssl`, and `native-tls` prevents accidental linkage to the C library being replaced. This is enforced at the dependency resolution level via `cargo deny check`, running in CI (Gate 10). If any transitive dependency pulls in these crates, the build fails. Alternative (a) was rejected because even test dependencies can influence the build graph. Alternative (c) was rejected because `libc` is a legitimate `-sys` dependency for the FFI crate. (AAP §0.6.4) | Transitive dependency chain may inadvertently introduce banned crates through deeply nested dependencies; developers must check ban compliance before adding any new dependency; some useful testing crates may depend on `openssl-sys` transitively | 2026-04-09 |

## Convention and Style Decisions

| # | Decision | Alternatives Considered | Rationale | Risks | Date |
|---|----------|------------------------|-----------|-------|------|
| C-1 | **Naming convention: idiomatic Rust `snake_case`** throughout, following Rust API Guidelines (RFC 430) | (a) Preserve original C naming (`SSL_CTX_new` → `ssl_ctx_new`); (b) Hybrid: C names in FFI crate, Rust names elsewhere; (c) PascalCase for all public types matching C typedefs | All Rust code follows RFC 430: `snake_case` for functions/methods/modules, `CamelCase` for types/traits, `SCREAMING_SNAKE_CASE` for constants. This means `SSL_CTX` → `SslCtx`, `EVP_MD_fetch` → `MessageDigest::fetch`, `OSSL_PROVIDER` → `Provider`. The FFI crate (`openssl-ffi`) preserves C naming via `#[no_mangle] pub extern "C" fn SSL_CTX_new(...)` for backward compatibility. Alternative (a) was rejected because it violates Rust conventions and triggers compiler warnings. (AAP §0.8.6) | Mapping confusion between C names (used in documentation, RFCs, error messages) and Rust names; developers familiar with OpenSSL C API must learn new names; `TRACEABILITY.md` provides the bidirectional mapping to mitigate this | 2026-04-09 |
| C-2 | **Documentation: `///` doc comments on every public item**, `//!` module docs on crate roots | (a) Minimal documentation, rely on code readability; (b) External documentation only (no inline docs); (c) Documentation only for complex functions | Every public item (function, struct, enum, trait, constant) carries a `///` doc comment. Crate root `lib.rs` files have `//!` module-level documentation explaining the crate's purpose, boundaries, and relationship to the C source. This supports `cargo doc --workspace` generation and is a prerequisite for the onboarding documentation requirement. Alternative (a) was rejected because a 572,000-LoC migration is not self-documenting. (AAP §0.8.6) | Documentation maintenance burden; risk of docs becoming stale as code evolves; doc comments on every item increases file size and potentially slows IDE parsing | 2026-04-09 |
| C-3 | **No `unwrap()` or `expect()` in library code** — only permitted in tests and CLI `main()` | (a) Allow `expect()` with descriptive messages in library code; (b) Allow `unwrap()` for cases provably safe (e.g., regex compilation); (c) No restrictions | Library crates (`openssl-common`, `openssl-crypto`, `openssl-ssl`, `openssl-provider`, `openssl-fips`) use `Result<T, E>` exclusively. `unwrap()` and `expect()` panics are inappropriate in a cryptographic library where callers must handle errors gracefully. Enforced via `clippy::unwrap_used = "warn"` and `clippy::expect_used = "warn"` in workspace lints. Tests and `openssl-cli::main()` may use `#[allow]` with justification. (AAP §0.8.6) | Increased verbosity from `?` propagation; some provably-infallible operations (e.g., `"constant".parse::<Regex>()`) require workarounds; error types must be defined even for truly unreachable error paths | 2026-04-09 |
